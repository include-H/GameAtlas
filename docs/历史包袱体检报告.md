# 历史包袱体检报告

更新时间：2026-03-24  
范围：`frontend/`、`backend/`、构建脚本与仓库产物

## 背景

本报告用于识别当前项目中的“历史包袱”（legacy debt），并给出可执行的治理优先级。  
目标不是一次性重写，而是降低后续迭代成本、减少回归风险、提升交付速度。

## 结论摘要（按优先级）

1. 前端超大组件职责过载（高优先级）
2. 数据库迁移兼容路径复杂（已治理完成）
3. 迁移 SQL 语句切分方式脆弱（高优先级）
4. 仓库跟踪二进制产物（中高优先级）
5. 前后端 DTO 存在兼容叠层（中优先级）
6. `any` 使用范围偏大（中优先级）
7. 自动化测试覆盖薄弱（中优先级）

## 详细发现

### 1) 前端超大组件职责过载

**现象**
- [`frontend/src/components/EditGameModal.vue`](/home/Hao/Game/frontend/src/components/EditGameModal.vue) 约 3773 行。
- 单组件内同时承担：表单状态、素材上传、Steam 数据拉取、标签创建、文件管理、提交流程编排等。

**风险**
- 变更影响面过大，回归概率上升。
- 新功能接入速度下降，排查问题成本高。
- 类型和状态耦合导致“修一处坏多处”。

**建议**
- 先按功能切 3~5 个 composable：`useGameFormCore`、`useGameAssets`、`useSteamImport`、`useTagSelection`、`useGameFilePaths`。
- 再拆 UI 子组件：媒体区、标签区、文件区。
- 用“外观层 + 领域层”结构，避免继续把逻辑堆进主组件。
- 补一层可复用样式约束：抽统一文本动作按钮 class（基于全局 `--color-text-*`），避免编辑页出现蓝色按钮与详情页白字体系不一致。


已完成
---

### 2) 数据库迁移兼容路径复杂（已治理完成）

**治理结果**
- 迁移实现已改为“按嵌入 SQL 文件名顺序执行 + `schema_migrations` 记录判重”，移除 `LegacyNames + Skip` 兼容桥接路径。
- 已删除历史兼容迁移 `000002 ~ 000012`，仅保留 [`backend/migrations/000001_initial_schema.sql`](/home/Hao/Game/backend/migrations/000001_initial_schema.sql) 作为当前基线。
- 迁移测试已切换为“新库全量执行 + 重复执行幂等性”校验：[`backend/internal/db/migrate_test.go`](/home/Hao/Game/backend/internal/db/migrate_test.go)。

**说明**
- 该治理方案适用于当前阶段（高强度开发、单团队使用、生产仅落后一版且无新增字段）。
- 后续迁移以“当前基线 + 增量迁移”为准，不再承诺更老版本自动升级。

---

### 3) 迁移 SQL 切分策略脆弱

**现象**
- [`backend/internal/db/migrate.go`](/home/Hao/Game/backend/internal/db/migrate.go#L296) 使用 `strings.Split(content, ";")` 分割 SQL。

**风险**
- 当语句内出现分号（复杂 SQL、触发器、字符串常量）时可能被误切分。
- 迁移失败形式隐蔽，定位困难。

**建议**
- 中期替换为更稳妥方案：按 migration 文件粒度执行，或引入 SQL parser / 更安全的 statement splitter。
- 短期增加防护：若检测到 trigger/BEGIN...END 结构，拒绝执行并给出明确错误。


已完成

- `release/` 目录忽略规则保持生效，继续通过 `build-release.sh` 或 CI 产出发布包。

已完成

### 5) DTO 兼容叠层导致映射复杂

**现象**
- 前端类型中同时存在多组历史兼容字段：  
  - `preview_video` / `preview_videos`  
  - `screenshots` / `screenshot_items`  
  - `file_path` / `file_paths`  
  见 [`frontend/src/services/types.ts`](/home/Hao/Game/frontend/src/services/types.ts)。
- 服务映射逻辑复杂：[`frontend/src/services/games.service.ts`](/home/Hao/Game/frontend/src/services/games.service.ts)。

**风险**
- 前后端字段语义不一致，产生边界 Bug。
- 新功能必须不断“补映射”，维护成本持续上升。

**建议**
- 定义单一 canonical DTO（例如统一为 `preview_videos`、`screenshot_items`、`files`）。
- 标记兼容字段为 deprecated，设置两阶段下线计划（写兼容 -> 读兼容 -> 删除）。

---

### 6) 类型系统存在 `any` 扩散

**现象**
- 多处出现 `any`（组件、视图、API 封装等），例如：  
  - [`frontend/src/components/EditGameModal.vue`](/home/Hao/Game/frontend/src/components/EditGameModal.vue)  
  - [`frontend/src/services/api.ts`](/home/Hao/Game/frontend/src/services/api.ts)  
  - [`frontend/src/views/GamesView.vue`](/home/Hao/Game/frontend/src/views/GamesView.vue)

**风险**
- 重构时缺少类型保护，线上问题更晚暴露。
- IDE/静态分析收益下降。

**建议**
- 先从高频入口收紧：`api.ts` 泛型、上传回调类型、表单 payload 类型。
- 逐步替换 `any -> unknown -> 精确类型`，避免一次性大改。

---

### 7) 自动化测试覆盖薄弱

**现象**
- 后端当前只有迁移测试文件：[`backend/internal/db/migrate_test.go`](/home/Hao/Game/backend/internal/db/migrate_test.go)。
- `go test ./...` 虽通过，但绝大多数包为 `[no test files]`。
- 前端 lint 通过，但缺业务单测/集成测试层。

**风险**
- 需求迭代缺少回归保护网。
- 代码越老越不敢改，债务持续累积。

**建议**
- 后端优先补：`services/games`、`repositories/games`、`handlers/games` 的关键路径测试。
- 前端优先补：`games.service` 映射逻辑和 `EditGameModal` 提交流程的核心行为测试。

## 推荐治理顺序（4 周最小闭环）

### 第 1 周：止损
- 清理仓库二进制跟踪（保留构建脚本产物路径）。
- 冻结 `EditGameModal` 新增功能，避免继续堆逻 辑。
- 新增迁移规范文档（命名、兼容保留期、退役规则）。

### 第 2 周：解耦
- 拆出 `EditGameModal` 的 composable（先逻辑后 UI）。
- 收敛 API/DTO 映射入口，建立 canonical DTO。

### 第 3 周：加护栏
- 后端补关键路径测试（至少覆盖创建/更新/查询主流程）。
- 前端补映射与提交流程测试。

### 第 4 周：迁移层稳态
- 优化迁移 SQL 执行策略，替换 `;` 直接切分方案。
- 回顾并清理可退役的 legacy 兼容桥。

## 验收标准（Done Criteria）

- `EditGameModal` 主文件行数显著下降，关键逻辑有独立模块。
- DTO 只保留一套主语义字段，兼容字段有明确下线时间。
- 后端核心业务包不再是大面积 `[no test files]`。
- 迁移框架对复杂 SQL 的容错策略明确且可测试。
- 仓库无跟踪二进制编译产物。
