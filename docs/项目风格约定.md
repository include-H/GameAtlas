# 项目风格约定（前后端）

更新时间：2026-03-24  
适用范围：`frontend/`、`backend/`

## 1. 目标

本约定用于统一协作风格，减少“同一问题多种写法”带来的维护成本。  
原则：先与现有代码兼容，再逐步收敛历史包袱。

## 2. 总体约定

- 优先可读性，其次才是“炫技”。
- 新代码必须可测试、可定位、可回滚。
- 新增逻辑不继续堆到超大文件里，优先拆分模块。
- 禁止把编译产物（二进制、打包目录）纳入版本库。

## 3. 前端风格（Vue 3 + TS + Arco）

### 3.1 文件与命名

- 组件使用 PascalCase：`GameDetailView.vue`、`EditGameModal.vue`。
- composable 使用 `useXxx` 命名，放 `src/composables/` 或 `src/hooks/`。
- service 按资源拆分，避免“全能 service”继续膨胀。

### 3.2 组件职责

- 单组件应聚焦单一页面/业务块。
- 超过约 800 行的组件默认进入拆分评审。
- 组件内同时出现“复杂表单 + 远程拉取 + 文件上传 + 提交流程编排”时，必须拆为 composable。

### 3.3 类型约束

- 禁止新增裸 `any`；历史 `any` 按 `any -> unknown -> 精确类型` 渐进替换。
- API 返回类型必须显式声明，不使用无约束泛型默认值。
- DTO 统一 canonical 字段，避免长期并存双字段语义（如 `x`/`x_list` 同时写读）。

### 3.4 状态与副作用

- “数据处理逻辑”优先放 composable，“视图绑定逻辑”留在组件。
- `watch` 只处理明确依赖，避免隐式联动链。
- 异步流程必须有 loading/error/finally 完整状态。

### 3.5 错误处理与反馈

- 对用户可见错误统一通过 UI 提示（如 `uiStore.addAlert`）。
- `console.error` 可保留在开发态，但关键链路需补充用户反馈与上下文信息。
- 提交流程错误信息要可定位（至少包含业务动作名）。

### 3.6 样式与可维护性

- 样式按组件局部作用域组织，避免全局污染。
- 复杂样式块使用语义化 class，不以结构层级硬编码。
- 视觉改动保持现有设计语言连续性，不跨页面“换体系”。

### 3.7 文本动作按钮规范

- 所有 `type="text"` 的“文本动作按钮”默认显式加 `app-text-action-btn`。
- 业务类（如 `xxx-btn`、`__action`）只用于布局/尺寸/间距，不承担颜色、hover、icon 颜色等外观职责。
- 禁止新增页面级按钮外观覆盖类（例如 `header-xxx-btn`、`version-xxx-btn` 这类“只改颜色”的类）。
- 危险态、成功态、警告态优先使用 Arco 状态能力（如 `status="danger"`），并复用 `app-text-action-btn` 的全局状态样式。
- 输入行右侧按钮统一模式：按钮类使用 `app-text-action-btn + 业务布局类`；布局细节仍放在业务类中维护。
- 迁移存量代码时，按“触达即治理”原则：改到的文本按钮同步切到 `app-text-action-btn`，并删除对应旧外观样式。

### 3.8 玻璃容器（Glass）规范

- 统一基线：容器玻璃效果以 `src/assets/style.css` 的 token 与全局类为准，不在页面内重复定义同语义值。
- 基础容器：使用 `app-glass-surface`（背景、边框、模糊、基础阴影）。
- 可交互容器：在基础容器上叠加 `app-glass-surface--interactive`（hover 边框与阴影）。
- Arco 容器（`a-card`/`a-modal`）默认走全局覆盖；仅允许在业务样式中补布局（padding、gap、height、grid），不重写玻璃核心属性。
- 禁止新增：随意写 `background: rgba(...) + backdrop-filter + border + box-shadow` 的“页面私有玻璃组合”；若确有需求，先补 token，再复用全局类。
- 允许例外：品牌化/沉浸式页面（如登录页）可局部偏离，但必须保持“同页内统一”，且在样式块注释说明“这是特例，不外溢到通用容器”。

## 4. 后端风格（Go + Gin + SQLX + SQLite）

### 4.1 分层边界

- `handler`：只做协议层（参数解析、状态码、响应格式）。
- `service`：业务规则与流程编排。
- `repository`：数据访问与 SQL。
- `domain`：领域对象与输入输出结构。

禁止跨层直接串用（例如 handler 直接写 SQL）。

### 4.2 后端允许做什么

- 允许在 `handler` 做：
  - 参数解析、鉴权门禁、HTTP 状态码映射、响应结构组装。
  - 极薄的协议适配，例如 query/path/body 到 service 输入结构的转换。
- 允许在 `service` 做：
  - 业务校验、trim/normalize、流程编排、跨 repository 聚合。
  - 文件系统与数据库的协调，但必须明确失败策略与补偿方式。
  - 领域规则计算，例如可见性、待处理项、聚合写入规则。
- 允许在 `repository` 做：
  - 只和存储有关的查询、更新、事务、行映射。
  - 为稳定读模型写清晰 SQL；可以有必要的只读聚合查询。
- 允许在 `domain` 做：
  - 稳定的领域对象、输入结构、只读结果结构。
  - 明确表达业务语义的枚举、常量、规则定义。

### 4.3 后端禁止做什么

- 禁止 `handler`：
  - 直接写 SQL、直接操作文件系统、直接拼业务规则。
  - 长期保存进程内业务状态作为正式能力的一部分。
  - 为单个接口发展独立“协议解释器”而不收敛到 service/domain。
- 禁止 `service`：
  - 把表名、join 表、列名作为字符串一路向下透传，制造伪通用抽象。
  - 为了“将来扩展”先加一层 wrapper/adapter/no-op 抽象。
  - 默默吞掉跨边界失败（例如 DB 成功、文件删除失败）而不记录后续处理路径。
- 禁止 `repository`：
  - 承担领域规则定义。
  - 在 SQL 中埋业务策略常量，导致 service 变成空壳。
  - 把多个不同用法硬塞进同一个巨型仓储文件继续膨胀。
- 禁止 `domain`：
  - 一个结构同时承担数据库行、列表 DTO、详情 DTO、统计容器多种职责。
  - 通过补假字段（如 `0 AS x`、`NULL AS y`）来迁就失真的结构。

### 4.4 原生与依赖原则

- 优先标准库、Gin、SQLX、SQLite 已有能力。
- 能直接用原生能力完成的，不新增第三方依赖。
- 能用明确函数解决的，不额外包“统一 helper”“兼容层”“适配器”。
- 禁止为了旧环境、旧接口、理论扩展性默认保留双写法或 fallback 分支。
- 新增依赖前先回答两个问题：
  - 当前仓库已有能力是否足够？
  - 这项依赖是否会长期存在，而不是只为一个局部问题服务？

### 4.5 错误与返回

- 仓储层返回底层错误，服务层做语义归一（如 `ErrValidation`、`ErrNotFound`）。
- handler 统一把服务错误映射为 HTTP 响应，不散落自定义分支。
- 错误文案要稳定，便于前端处理。
- 禁止同一类错误在多个 handler 中各写一套映射分支。
- 禁止把“本应暴露为明确问题”的失败降级成日志后静默跳过，除非这是显式设计并有后续处理路径。

### 4.6 SQL 与迁移

- 迁移采用“单基线 + 增量”模式：基线文件为 [`backend/migrations/000001_initial_schema.sql`](/home/Hao/Game/Game/backend/migrations/000001_initial_schema.sql)。
- 新增迁移文件必须使用递增编号与语义化命名（如 `000013_xxx.sql`），且提交后不修改既有编号与语义。
- 迁移执行器按文件名顺序执行，基于 `schema_migrations.name` 判重；禁止再引入 `LegacyNames/Skip` 一类隐式兼容分支。
- 复杂升级场景优先写显式增量 SQL，不在执行器中堆“按表结构猜测”的跳过逻辑。
- 如需“破坏式重置基线”（squash migrations），必须在迭代说明中明确窗口与回滚方案后统一执行。
- 禁止把一次性数据修复长期挂在应用启动路径里反复执行。
- 禁止为适配失真结构在多个查询里重复手写同一批占位列与聚合列。
- 复杂查询允许存在，但必须服务于稳定读模型；若同类查询开始复制扩散，应优先收敛。

### 4.7 数据结构

- 领域模型字段命名保持前后端一致语义。
- 布尔/枚举值统一在服务层校验，不把非法值放进仓储层处理。
- 输入结构在进入 repository 前必须完成 trim/normalize。
- 精确时间戳和模糊业务日期不要长期混作同一语义字符串。
- 用户可管理资源与内部附属数据要区分清楚，禁止一边开放创建、一边在后台自动按“垃圾数据”清理。

### 4.8 文件与外部系统边界

- 路径校验统一走共享边界，禁止在 route/service 各写一套近似规则。
- 文件删除、下载、素材落盘、脚本生成这类跨边界动作，必须明确：
  - 谁负责调用。
  - 失败后怎么处理。
  - 是否允许部分成功。
- 平台特化逻辑（如 Windows 启动脚本）不得继续侵入通用文件服务；新增平台能力优先独立边界。
- 文档中的访问控制、下载策略、去重策略必须和实现一致；文档不是愿景列表。

### 4.9 新增迁移操作模板

1. 新建迁移文件  
   在 `backend/migrations/` 下创建下一个编号文件，例如 `000013_add_games_xxx.sql`。
2. 编写前向迁移 SQL  
   只写“从当前基线继续前进”的变更，不写 `LegacyNames/Skip` 兼容分支，不改旧文件。
3. 本地验证迁移  
   优先运行 `cd backend && bash check.sh`；若单独执行命令，至少保证 `go test ./...` 覆盖“新库执行成功 + 重复执行幂等”。
4. 提交说明  
   在 PR 描述中写清楚：迁移目的、影响表/索引、是否存在回滚策略。

## 5. 测试与质量门槛

### 5.1 前端

- 至少覆盖：service 映射逻辑、关键表单提交流程、核心筛选/排序逻辑。
- bug 修复必须附带对应测试或可复现用例说明。

### 5.2 后端

- 新增业务逻辑至少补 service 或 repository 级测试其一。
- 影响接口行为的改动应补 handler 层回归测试。
- 迁移变更必须补迁移测试，至少覆盖“新库执行成功 + 重复执行幂等”。
- 测试优先覆盖行为边界，不绑定实现细节。
- 文档中写成正式能力的规则，必须有对应测试支撑。
- 高风险链路优先补测试：
  - 删除与清理。
  - 鉴权与访问控制。
  - 文件边界与下载。
  - 聚合更新与事务一致性。

### 5.3 合并前检查

- 前端：`npm run lint` 通过。
- 后端：`cd backend && bash check.sh` 通过。
- 不允许引入新的已知高风险债务（超大文件继续膨胀、无类型新增 `any`、迁移无测试）。

## 6. 渐进治理策略（与当前代码兼容）

- 存量代码不做一次性重写，采用“触达即治理”：
  - 修改某模块时，顺带清理该模块内最明显风格债务。
  - 新代码严格执行新约定，旧代码按迭代逐步靠拢。
- 对高风险文件（如超大组件）采用“先拆逻辑后拆视图”的低风险路径。

## 7. 推荐在 PR 模板增加的检查项

- 是否新增了 `any`？若有，是否给出替代计划？
- 是否扩大了单文件职责？是否可拆分？
- 是否引入了新的 DTO 双语义字段？
- 是否补充了对应层级测试？
- 是否影响迁移与历史兼容路径？
