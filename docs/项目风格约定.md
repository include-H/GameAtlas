# 项目风格约定（前后端）

更新时间：2026-03-24  
适用范围：`frontend/`、`backend/`

## 1. 目标

本约定用于统一协作风格，减少“同一问题多种写法”带来的维护成本。  
原则：先与现有代码兼容，再逐步收敛历史包袱。

## 2. 总体约定

- 优先可读性，其次才是“炫技”。
- 新代码必须可测试、可定位、可回滚。
- 新增逻辑不继续堆到超大文件里，优先拆分模块。
- 禁止把编译产物（二进制、打包目录）纳入版本库。

## 3. 前端风格（Vue 3 + TS + Arco）

### 3.1 文件与命名

- 组件使用 PascalCase：`GameDetailView.vue`、`EditGameModal.vue`。
- composable 使用 `useXxx` 命名，放 `src/composables/` 或 `src/hooks/`。
- service 按资源拆分，避免“全能 service”继续膨胀。

### 3.2 组件职责

- 单组件应聚焦单一页面/业务块。
- 超过约 800 行的组件默认进入拆分评审。
- 组件内同时出现“复杂表单 + 远程拉取 + 文件上传 + 提交流程编排”时，必须拆为 composable。

### 3.3 类型约束

- 禁止新增裸 `any`；历史 `any` 按 `any -> unknown -> 精确类型` 渐进替换。
- API 返回类型必须显式声明，不使用无约束泛型默认值。
- DTO 统一 canonical 字段，避免长期并存双字段语义（如 `x`/`x_list` 同时写读）。

### 3.4 状态与副作用

- “数据处理逻辑”优先放 composable，“视图绑定逻辑”留在组件。
- `watch` 只处理明确依赖，避免隐式联动链。
- 异步流程必须有 loading/error/finally 完整状态。

### 3.5 错误处理与反馈

- 对用户可见错误统一通过 UI 提示（如 `uiStore.addAlert`）。
- `console.error` 可保留在开发态，但关键链路需补充用户反馈与上下文信息。
- 提交流程错误信息要可定位（至少包含业务动作名）。

### 3.6 样式与可维护性

- 样式按组件局部作用域组织，避免全局污染。
- 复杂样式块使用语义化 class，不以结构层级硬编码。
- 视觉改动保持现有设计语言连续性，不跨页面“换体系”。

## 4. 后端风格（Go + Gin + SQLX + SQLite）

### 4.1 分层边界

- `handler`：只做协议层（参数解析、状态码、响应格式）。
- `service`：业务规则与流程编排。
- `repository`：数据访问与 SQL。
- `domain`：领域对象与输入输出结构。

禁止跨层直接串用（例如 handler 直接写 SQL）。

### 4.2 错误与返回

- 仓储层返回底层错误，服务层做语义归一（如 `ErrValidation`、`ErrNotFound`）。
- handler 统一把服务错误映射为 HTTP 响应，不散落自定义分支。
- 错误文案要稳定，便于前端处理。

### 4.3 SQL 与迁移

- 迁移文件命名保持递增、有语义，不随意改历史编号。
- 新迁移优先“前向演进”，少做隐式兼容魔法。
- 若必须保留 legacy 兼容逻辑，需写明退役条件。
- 避免依赖脆弱 SQL 切分策略，复杂 SQL（trigger/BEGIN...END）需显式验证。

### 4.4 数据结构

- 领域模型字段命名保持前后端一致语义。
- 布尔/枚举值统一在服务层校验，不把非法值放进仓储层处理。
- 输入结构在进入 repository 前必须完成 trim/normalize。

## 5. 测试与质量门槛

### 5.1 前端

- 至少覆盖：service 映射逻辑、关键表单提交流程、核心筛选/排序逻辑。
- bug 修复必须附带对应测试或可复现用例说明。

### 5.2 后端

- 新增业务逻辑至少补 service 或 repository 级测试其一。
- 影响接口行为的改动应补 handler 层回归测试。
- 迁移变更必须补迁移测试（尤其是旧库升级路径）。

### 5.3 合并前检查

- 前端：`npm run lint` 通过。
- 后端：`go test ./...` 通过。
- 不允许引入新的已知高风险债务（超大文件继续膨胀、无类型新增 `any`、迁移无测试）。

## 6. 渐进治理策略（与当前代码兼容）

- 存量代码不做一次性重写，采用“触达即治理”：
  - 修改某模块时，顺带清理该模块内最明显风格债务。
  - 新代码严格执行新约定，旧代码按迭代逐步靠拢。
- 对高风险文件（如超大组件）采用“先拆逻辑后拆视图”的低风险路径。

## 7. 推荐在 PR 模板增加的检查项

- 是否新增了 `any`？若有，是否给出替代计划？
- 是否扩大了单文件职责？是否可拆分？
- 是否引入了新的 DTO 双语义字段？
- 是否补充了对应层级测试？
- 是否影响迁移与历史兼容路径？
